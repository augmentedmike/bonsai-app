# Daily Learning — Developer — 2026-02-25

## Topic: Intentional Error Boundaries — Designing for Failure First

Most developers think about the happy path first and bolt on error handling afterward. In 2025 and beyond, the best engineers flip this around: they design the failure surface before writing a single line of business logic. This "failure-first" mindset produces dramatically more resilient software with far less debugging time.

**What it means in practice**

When starting a new feature, ask: *What are the 5 ways this can go wrong?* Write that list down before touching the keyboard. Map out: network failures, malformed input, race conditions, third-party API timeouts, and user abuse cases. Only after that list exists should you start coding.

In React (and most modern UI frameworks), this maps directly to **Error Boundaries** — components that catch rendering errors and present graceful fallback UI instead of crashing the entire tree. But the mindset extends far beyond React:

- **Backend**: Wrap every external call (DB, Redis, third-party API) in a circuit breaker pattern. Libraries like `opossum` (Node) or `resilience4j` (JVM) make this concrete.
- **API design**: Return structured error objects, not raw exceptions. A response like `{ "error": { "code": "RATE_LIMITED", "retryAfter": 30 } }` is infinitely more useful to clients than a bare 500.
- **Edge functions / serverless**: Every cold-start or timeout is a failure mode. Set explicit timeouts shorter than the platform max so YOU control the failure, not the platform.

**The shift from defensive to intentional**

Defensive programming is reactive — you handle errors when you remember to. Intentional failure design is proactive — errors are first-class citizens in your architecture from day one.

A concrete habit: before opening a PR, add a `## Failure Modes` section to your PR description. List what you've handled and what you've consciously chosen to leave for the caller. This discipline forces clarity during code review and creates a paper trail for future debugging.

**The AI coding context**

With LLM-assisted development now standard, there's a new failure mode: AI-generated code often skips error handling because it optimizes for the happy path to look clean and complete. When reviewing AI-generated code, your first pass should specifically scan for missing error handling — timeouts, null checks, retry logic, and graceful degradation. Make "what happens when this fails?" your default review question.

**Tools to know**

- `zod` / `valibot` — runtime schema validation; reject bad input at the boundary
- `opossum` — Node.js circuit breaker
- Sentry + source maps — make production errors actionable, not just logged
- OpenTelemetry — trace failures across distributed systems

The developers who ship stable software in 2026 aren't smarter — they're more systematic about failure. Build the habit now.

## Key Takeaway
Design your failure surface before your happy path — intentional error handling is the highest-leverage habit separating senior from junior engineers in 2026.
